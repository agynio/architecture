## Delta

### Metering Service

- The Metering Service does not exist. No `agynio/metering` repository, no `MeteringService.Record` gRPC endpoint, no storage.
- The `metering/v1/metering.proto` definition does not exist in `agynio/api`.
- The Gateway does not expose a usage query API.

### LLM Proxy

- The LLM Proxy does not emit usage records to the Metering Service after completed calls.
- Token counts are not extracted from provider responses (non-streaming `usage` object or final SSE event).
- The `x-agyn-thread-id` header is not read or forwarded as a label.

### Runners Service

- No volume state tracking. `CreateVolume`, `UpdateVolume`, `GetVolume`, `ListVolumes`, `ListVolumesByThread` methods do not exist.
- The Volume resource does not exist — no `volumes` table, no volume status lifecycle (`provisioning → active → deprovisioning → deleted`), no `removed_at` or `last_metering_sampled_at` fields.
- `ListWorkloads` and `ListVolumes` do not support `pending_sample` or `status_in` filters.
- `BatchUpdateWorkloadSampledAt` and `BatchUpdateVolumeSampledAt` bulk update methods do not exist.
- The Workload resource does not have `instance_id`, `status`, `removed_at`, or `last_metering_sampled_at` fields.
- `CreateWorkload` is not called before `StartWorkload` — workload records are not pre-created with `status=starting`.
- Workload state lifecycle is not tracked end-to-end: the Orchestrator does not call `UpdateWorkload` to transition through `starting → running → stopping → stopped`, does not set `removed_at` after stop, and does not retain workload records for audit after removal.
- `agynd` does not call `TouchWorkload` to update `last_activity_at`. Idle timeout enforcement based on `last_activity_at` is not implemented.

### Runner / k8s-runner

- `StartWorkload` does not embed `workload_key` and `volume_key` labels in the workload spec, and does not return `instance_id`.
- `ListWorkloads` does not return the `workload_key` label alongside `instance_id`.
- `ListVolumes` does not exist. PVCs are not listed or exposed via the Runner gRPC API.

### Resource Definitions

- The Volume resource does not have a `ttl` field. Volume TTL enforcement is not implemented.

### Agents Orchestrator

- The Orchestrator does not generate `workload_key` and `volume_key` UUIDs or embed them as labels in the workload spec.
- The Orchestrator does not pre-create workload and volume records in the Runners service before calling `StartWorkload`.
- The agent stop flow does not set `status=stopping` before calling `StopWorkload`, and does not set `removed_at` after `StopWorkload` succeeds.
- The Orchestrator has no workload reconciliation loop — divergence between Runners service records and actual runner state is not detected or corrected.
- The Orchestrator has no volume reconciliation loop — PVC lifecycle and TTL enforcement are not reconciled.
- The Orchestrator has no metering sampling loop. No compute usage records (CORE_SECONDS, GB_SECONDS for RAM) are emitted for running workloads.
- No storage usage records (GB_SECONDS for persistent volumes) are emitted.

### Threads Service

- The Threads service does not emit usage records to the Metering Service on thread or message creation.

### Console

- No usage dashboard exists. Org admins have no visibility into LLM token usage, compute, storage, or platform entity counts.

## Acceptance Signal

- The Metering Service accepts `Record` calls and stores usage events in a monthly-partitioned `usage_events` table, deduplicated by `UNIQUE(idempotency_key, month)`.
- The LLM Proxy emits token and request COUNT records after each completed call. Records appear in the Metering Service within one reconciliation cycle.
- The Runners service tracks workload lifecycle: workloads are pre-created with `status=starting`, transition through `running → stopping → stopped`, retain `removed_at` after removal, and are retained as audit history.
- `agynd` calls `TouchWorkload` every 10 seconds while active; idle timeout enforcement stops workloads where `now - last_activity_at > idle_timeout`.
- The Runners service tracks volume lifecycle: volumes are created with `status=provisioning` before `StartWorkload` and progress through `active → deprovisioning → deleted`.
- The Orchestrator workload reconciliation loop detects and corrects divergence between Runners service records and actual runner state within one reconciliation cycle.
- The Orchestrator volume reconciliation loop detects PVC presence/absence and enforces TTL on persistent volumes.
- The Orchestrator emits CORE_SECONDS and GB_SECONDS records for all running workloads and persistent volumes every 60 seconds.
- The Threads service emits COUNT records on thread and message creation.
- The console usage dashboard displays LLM token usage (daily chart, by consumer, by model), compute usage (daily chart, by consumer), storage usage (daily chart, by consumer), and platform counts (daily chart) for the selected time range.

## Notes

- Billing aggregation over billing periods is a future concern. This change covers observability only.
- Compute and storage usage is based on allocated resources, not measured consumption.
