# Inline Media

## Purpose

Conversations display media — images, video, and audio — inline within messages. Users see visual content without leaving the conversation. All media is loaded through the platform's media proxy for security and privacy.

## Sources

Media appears in messages from two sources:

1. **External URLs** — agents respond with markdown containing image, video, or audio links to public URLs. Example: `![chart](https://example.com/chart.png)`.
2. **Platform files** — files uploaded to the platform or generated by agents, referenced via `agyn://file/<id>` URIs.

## Images

Images render inline within the message body at a constrained display width. Large images are downsampled for inline display to reduce bandwidth.

Each image includes a **download link** that opens the original full-resolution file.

## Video

Videos render as an inline `<video>` player within the message body. The player supports:

- Play/pause.
- Seeking (scrubbing the timeline).
- Volume control.
- Fullscreen toggle.

Each video includes a **download link** for the original file.

## Audio

Audio renders as an inline `<audio>` player within the message body. The player supports:

- Play/pause.
- Seeking.
- Volume control.

Each audio clip includes a **download link** for the original file.

## Markdown Syntax

The chat composer and message renderer support standard markdown media syntax:

| Type | Syntax |
|------|--------|
| Image | `![alt text](url)` |
| Video | `![video](url)` rendered as `<video>` when the content type is `video/*` |
| Audio | `![audio](url)` rendered as `<audio>` when the content type is `audio/*` |

The renderer determines the media type from the proxy response's `Content-Type` header and selects the appropriate display element.

## Security

The user's browser never contacts third-party servers directly to load media. All media — both external URLs and platform files — is loaded through the platform's [media proxy](../../architecture/media-proxy.md). This prevents:

- IP address leakage to third parties.
- Tracking pixels and cookies.
- Mixed-content warnings.

## Authentication

Media proxy requests require the same authentication as all other platform API calls. A **Service Worker** registered by the chat app intercepts all requests to the media proxy endpoint and injects the user's authentication token. This is transparent to the user — media loads as if it were public.

The Service Worker approach means that `<img>`, `<video>`, and `<audio>` elements use plain `src` attributes pointing to the media proxy URL. The browser handles caching, range requests (for video/audio seeking), and progressive loading natively.

## Caching

The browser caches media responses using standard HTTP cache headers returned by the media proxy. Scrolling back to a previously viewed message shows the media instantly without re-fetching. No application-level caching is required.

## Loading States

- While media is loading, a placeholder (shimmer or skeleton) is shown at the expected dimensions.
- If loading fails (network error, 4xx/5xx from the proxy), a fallback is shown with the alt text and a retry action.

## Constraints

- Media rendering depends on browser support for the media format. Unsupported formats fall back to a download link.
- The media proxy enforces a maximum file size. Media exceeding the limit is not rendered inline — a download link is shown instead.
