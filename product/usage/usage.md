# Usage

## Purpose

Usage gives org admins visibility into platform resource consumption across the organization — LLM capacity, compute, storage, and platform activity — broken down by time, agent, and model.

## Access

Usage is a sidebar section available to organization owners and cluster admins. It is scoped to the selected organization.

## User Stories

- As an org admin, I want to see total LLM token usage across my organization so I can monitor spend.
- As an org admin, I want to see LLM usage broken down by consumer and by model so I can identify heavy consumers and expensive models.
- As an org admin, I want to see what portion of my input tokens were served from the provider's cache so I can assess caching efficiency.
- As an org admin, I want to see request success and failure counts so I can detect reliability issues.
- As an org admin, I want to see compute and storage usage per agent so I can understand infrastructure consumption.
- As an org admin, I want to filter by time range so I can compare usage across periods.
- As an org admin, I want data to reflect the current state when I load the page.

## Time Range

A time range selector at the top of the page controls the window for all metrics. Changing the range reloads all sections.

| Option | Description | Chart grouping interval |
|--------|-------------|-------------------------|
| Last 24 hours | Rolling 24-hour window | 5 minutes |
| Last 7 days | Rolling 7-day window | 1 hour |
| Last 30 days | Rolling 30-day window | 6 hours |
| Custom | User-specified start and end date | Auto-selected to keep ~50–300 buckets: ≤ 24h → 5m, ≤ 7d → 1h, ≤ 30d → 6h, longer → 1d |

Time-series charts use these grouping intervals rather than daily buckets so short ranges show meaningful detail.

Data reflects the state at page load. There are no live updates — reload to see changes.

## Chart Style

Charts use a neon palette that renders cleanly on both light and dark themes, instead of the monochrome default. Each data series has a fixed color so a given metric (e.g., input tokens) looks the same across all charts on the page.

| Series | Color |
|--------|-------|
| Input tokens (fresh) | Cyan (`#00E5FF`) |
| Cached tokens | Magenta (`#FF4DD2`) |
| Output tokens | Lime (`#39FF88`) |

| RAM | Violet (`#B388FF`) |

| Messages | Lime (`#39FF88`) |

## Layout

The page is divided into four sections displayed vertically as a scrollable page: **LLM**, **Compute**, **Storage**, and **Platform**. Each section shows summary metric cards above its charts.

## LLM

Tracks token consumption and request outcomes for all LLM calls made by agents, users, and apps in the organization.

**Summary cards:**

| Metric | Description |
|--------|-------------|
| Input tokens | Total tokens in prompts sent to models (includes cached) |
| Cached tokens | Subset of input tokens served from the provider's prompt cache. Shown as `N of M input` to make the relationship explicit (e.g., "800 of 1,000 input") |
| Output tokens | Tokens generated by models |
| Successful requests | LLM calls that completed successfully |
| Failed requests | LLM calls that returned an error |

**Charts:**

| Chart | Description |
|-------|-------------|
| Tokens over time | Bar chart — tokens per bucket over the selected range, using the [grouping interval](#time-range) for the chosen range. Each bucket shows two stacked bars side by side: **input** (split into *cached* and *fresh input* — cached is a subset of input, not additive) and **output** |
| Top consumers | Horizontal bar chart — total tokens (input + output) per identity (agent, user, or app), sorted descending. Each bar is labeled with the consumer's display name — agent name for agents, `@nickname` (or full name) for users, app name for apps — never the raw identity ID. Hovering reveals the identity type and ID for disambiguation |
| By model | Horizontal bar chart — total tokens per model, sorted descending. Bars are labeled with the model's internal name |

**Empty state:** "No LLM usage in this period."

## Compute

Tracks CPU and RAM allocated to agent workloads. Values are allocation-based, not measured consumption — they reflect the resources reserved for workloads, not actual CPU/RAM utilization.

**Summary cards:**

| Metric | Description |
|--------|-------------|
| CPU-core-hours | Allocated CPU cores × hours across all workloads. Underlying values are core-seconds; the summary renders in hours for readability (`core-seconds / 3600`). Values below 1 hour show one decimal (e.g., `0.3 core-hours`) |
| RAM-GB-hours | Allocated RAM (GB) × hours across all workloads. Underlying values are GB-seconds; rendered in hours (`GB-seconds / 3600`) |

**Charts:**

| Chart | Description |
|-------|-------------|
| Usage over time | Bar chart — CPU-core-hours and RAM-GB-hours per bucket, using the [grouping interval](#time-range) for the chosen range. Two series shown side by side |
| Top agents | Horizontal bar chart — total CPU-core-hours and RAM-GB-hours per agent, sorted descending. Bars are labeled with the agent name, not the identity ID |

**Empty state:** "No compute usage in this period."

## Storage

Tracks persistent volume capacity allocated to agents. Values are allocation-based — they reflect the size of volumes attached to agents, not actual disk utilization.

**Summary cards:**

| Metric | Description |
|--------|-------------|
| Storage-GB-hours | Allocated volume size (GB) × hours across all persistent volumes. Underlying values are GB-seconds; rendered in hours (`GB-seconds / 3600`) |

**Charts:**

| Chart | Description |
|-------|-------------|
| Usage over time | Bar chart — storage-GB-hours per bucket, using the [grouping interval](#time-range) for the chosen range |
| Top agents | Horizontal bar chart — total storage-GB-hours per agent, sorted descending. Bars are labeled with the agent name, not the identity ID |

**Empty state:** "No storage usage in this period."

## Platform

Tracks creation of platform entities — thread and message volume.

**Summary cards:**

| Metric | Description |
|--------|-------------|
| Threads created | Threads created in the selected period |
| Messages sent | Messages sent in the selected period |

**Charts:**

| Chart | Description |
|-------|-------------|
| Activity over time | Bar chart — thread and message counts per bucket, using the [grouping interval](#time-range) for the chosen range. Two series shown side by side |

**Empty state:** "No platform activity in this period."

## Constraints

- Compute and storage values are allocation-based, not measured utilization.
- Data reflects the state at page load; there are no live updates.
- All metrics are scoped to the organization — cross-org views are not available.
